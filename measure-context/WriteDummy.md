あなたはClaude Code上で動作している。これからトークン消費計測用のサンプル資料一式を生成し、同時に「書き込み側トークン」を計測する。後続の「読み込み計測セッション」は別セッションで独立実行されるため、このセッションは(A)成果物のファイル書き出しと(B)書き込みトークンの計測の2点に責任を持つ。読み込みトークンの計測は行わない(別セッションの責務)。推測や捏造をせず、不明点や測定不能箇所があれば止めて報告すること。

# トークナイザ(全計測共通・APIキー不要)

トークン計測には Python の tiktoken を用いる(pip install tiktoken)。エンコーディングは cl100k_base を使用する。これはAnthropic本番トークナイザとは数え方が異なるため、得られる絶対値は近似値である。本計測の主目的は .txt を基準(1.00)とした形式間の相対倍率の比較であり、近似トークナイザでこの目的は達成できる。レポートには必ず「tiktoken cl100k_base による近似値」と明記すること。Anthropic公式 count_tokens API は使わない。

# 目的

同一の意味内容を持つ「車載ソフト仕様の技術検討資料」を複数のファイル形式で生成し、各形式を生成する際にClaudeが出力した内容のトークン数(=書き込みコンテキスト)を実測する。形式ごとのトークン特性の差を正しく測るため、内容は全形式で完全に一致させること。

# 題材(固定)

車線維持支援システム LKAS (Lane Keeping Assist System) のソフトウェア仕様検討資料。
含めるべき内容:

- システム概要と前提(対象車種クラス、動作速度域、センサ構成: 前方カメラ/操舵角センサ/ヨーレートセンサ等)
- 機能要求(車線検出、逸脱判定、操舵トルク介入、ドライバオーバーライド、警報)
- 状態遷移(OFF / STANDBY / ACTIVE / OVERRIDE / FAULT)
- 主要シーケンス(車線逸脱検知から操舵介入までの流れ)
- 処理フロー(1制御周期内の処理フローチャート)
- 機能安全に関する考慮(ISO 26262を意識したフォールト検出とフェールセーフの一般的記述。規格条文は引用しない)
- 主要パラメータ表とインタフェース表

# 分量(厳守)

- 総情報量は「A4用紙10枚相当」とする。
- うち情報量の50%を図表が、50%を本文テキストが担う構成とする。
- 本文テキスト量の目安: A4約5枚相当(日本語で概ね4000〜6000字程度)。図表は下記図表群でおおむね残り50%の情報を担うよう設計する。

# 図表の構成(全形式で同一の図・表を表現する)

以下を必ず含め、形式ごとに「その形式ネイティブの表現方法」で同じ図を描く:

1. システム構成図(ブロック図)
2. 状態遷移図(OFF/STANDBY/ACTIVE/OVERRIDE/FAULT)
3. シーケンス図(逸脱検知→操舵介入)
4. 処理フローチャート(1制御周期)
5. パラメータ表(10行以上)
6. インタフェース表(10行以上)
7. 画面キャプチャ 2枚(ダミー生成。後述)

# 画面キャプチャ(ダミー生成)

- capture1.png と capture2.png を生成する。内容は車載HMIの模擬画面(例: LKAS作動状態表示、警報表示)。matplotlib等で擬似ダッシュボードを描けばよい。実在画像は使わない。
- 解像度は2枚とも 1280x720 px に固定し、生成後に実ファイルの幅・高さをコードで確認してログ出力すること(書き込み画像トークン算出に必要)。

# 形式ごとの図の表現規則(倍率の本質)

- .txt : シーケンス図・フローチャート・状態遷移図・構成図はアスキーアートで描く。表はプレーンテキスト整形表。画像はファイル参照(`[画像: capture1.png]` のようにパス文字列のみ。実体は埋め込まない)。
- .md : 図はmermaid(flowchart/stateDiagram/sequenceDiagram)。表はMarkdown表。画像はファイル参照(`![説明](capture1.png)`)。
- .sdoc : StrictDoc形式(https://github.com/strictdoc-project/strictdoc の文法。[DOCUMENT][SECTION][REQUIREMENT]等のタグ付きテキスト。レンダリング後HTMLではない)。STATEMENT等のフィールド内にRST記法でmermaidをコードブロックとしてテキスト埋め込みする。表はRST表。画像はRST参照(`.. image:: capture1.png`)。pip install strictdoc を行い、生成した.sdocが `strictdoc export` でエラーなく通ることを検証する。通らなければ文法を修正する。
- .drawio : 図は全てmxGraph XML(drawioネイティブ)のオートシェイプ・コネクタで描く。構成図・状態遷移図・フロー・シーケンスを図形配置する。表は図形ベースまたはテキスト。
- .html / .xml : HTMLで本文を記述し、図はインラインSVGまたはmermaidコードブロックで表現。表は<table>。画像はファイル参照(<img src="capture1.png">)。
- Excel (.xlsx) : openpyxl使用。本文・表をセルに格納し、図はオートシェイプ(図形)で描画。画面キャプチャ2枚は画像実体としてシートに埋め込む(add_imageで実バイト格納=コピペ貼り付け相当)。
- Word (.docx) : python-docx使用。本文・見出し・表を構成し、図はオートシェイプ相当の図形で描く。画面キャプチャ2枚は画像実体として埋め込む(add_picture)。
- PowerPoint (.pptx) : python-pptx使用。各図をオートシェイプで作図しスライド構成にする。画面キャプチャ2枚は画像実体として埋め込む(add_picture)。

# 画像の埋め込み方針(厳守)

- txt / md / sdoc / drawio / html : 画像はパス参照のみ。実体は含めない。
- Office系(xlsx / docx / pptx) : 画面キャプチャ2枚を画像実体として埋め込む(コピペ貼り付け相当)。

# 出力先

全成果物を ./samples/ に出力。共通ベース名で揃える:
spec_lkas.txt / spec_lkas.md / spec_lkas.sdoc / spec_lkas.drawio / spec_lkas.html / spec_lkas.xlsx / spec_lkas.docx / spec_lkas.pptx
画像 capture1.png / capture2.png も ./samples/ に置く。

# 書き込みトークンの計測(このセッションの責務)

各ファイルを生成し終えた直後に、そのファイルに書き込んだ内容のトークン数を tiktoken(cl100k_base)で計測する。これが書き込みコンテキストである。

- 計測対象:
  - txt/md/sdoc/drawio/html/xml : ファイルに書き込んだテキスト全体を tiktoken でカウント。
  - xlsx/docx/pptx : 2通りを区別して両方記録する。
    (a) 論理内容ベース(主指標) = セル/段落/図形に書き込んだテキスト値を全て集めてテキスト化しカウント。他形式と公平に比較するための値。
    (b) 生成コードベース(参考値) = ファイル生成に用いたPythonソースをカウント。
- 書き込み画像トークン: 画像実体を埋め込む xlsx/docx/pptx は、埋め込み画像2枚分を (width×height)/750 で算出して加算する。txt/md/sdoc/drawio/html はパス参照のみのため0。((width×height)/750 はAnthropic公式の画像トークン概算式)
- 各形式について「書き込みテキスト由来トークン」「書き込み画像由来トークン」「書き込み合計」「.txt合計を1.00とした倍率」を表に記録する。
- 計測結果を ./samples/write_tokens.json に機械可読な形で保存する(後続セッションが読み込んでレポートに転記するため。各形式名をキーに、text_tokens/image_tokens/total/ratio を格納)。

# 完了時の報告

1. ./samples/ 配下の全ファイルを一覧表示し、各バイトサイズを示す。
2. capture1.png / capture2.png の実解像度(幅×高さ)を示す。
3. 上記「書き込みトークン」計測結果の表(内訳と.txt基準倍率付き)。
4. 各形式が全形式で同一内容になっていることを自己チェックし、内容差分があれば報告。
5. .sdocがstrictdocで検証通過したかを報告。
6. 測定環境(tiktokenのバージョン、各生成ライブラリのバージョン)。
7. write_tokens.json を書き出したことの確認。
8. 推測で埋めた箇所が無いことの確認、有る場合は明示。

注意: 読み込みトークンの計測はこのセッションでは行わない(別セッションの責務)。トークン値はすべて tiktoken による近似値である旨をレポートに明記すること。
